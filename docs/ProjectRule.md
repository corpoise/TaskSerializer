# ProjectRule

## 프로젝트 작성 지침

1. 명시적으로 요청되지 않은 기능, 버튼, UI 요소, 메서드는 추가하지 않는다.

2. 신규 솔루션 또는 프로젝트를 처음 생성할 때, 작업을 시작하기 전에 솔루션명과 프로젝트명을 먼저 합의한다.

3. 코드를 결과물로 제공하기 전에 반드시 아래 순서를 따른다:
   - `.editorconfig` 파일이 없으면 프로젝트 코딩 규칙에 맞게 생성한다.
   - `dotnet build` 로 빌드 성공 여부를 확인한다.
   - `dotnet format --verify-no-changes` 로 정적 검사를 수행한다.
   - 오류나 경고가 있으면 수정 후 위 과정을 다시 반복한다.
   - 모든 검사를 통과한 경우에만 결과물을 제공한다.

4. 결과물은 발생할 수 있는 이슈, 버그, 예외가 최대한 제거된 상태로 작성한다. 이를 위해 기능적 변경이나 작업 방향 변경이 필요한 경우, 변경 필요성과 영향을 설명하고 협의 후 진행한다.

5. 컨벤션 또는 규칙 문서에 새로운 규칙을 추가하거나 기존 규칙을 변경 또는 삭제해야 하는 경우, 변경 전에 반드시 변경 대상과 변경 내용을 설명하고 승인을 받는다.
   - 명시적 승인 없이 규칙을 추가, 변경, 삭제하지 않는다.
   - 승인받은 범위 안에서만 규칙 문서를 수정한다.
   - 승인 후 수정한 규칙이 현재 작업 중인 대상 프로젝트에 영향을 주는지 확인한다.

6. 컨벤션에 새로운 규칙이 추가되거나 기존 규칙이 수정되는 경우, 현재 작업 중인 대상 프로젝트에 변경된 컨벤션을 다시 적용한다.
   - 변경된 규칙에 영향을 받는 코드 후보를 먼저 검색한다.
   - 수정 대상과 수정 방향을 설명하고 승인받는다.
   - 승인 후 코드를 수정한다.
   - `dotnet build` 로 빌드 성공 여부를 확인한다.
   - `dotnet format --verify-no-changes` 로 정적 검사를 수행한다.
   - 오류나 경고가 있으면 수정 후 위 과정을 다시 반복한다.


## 테스트 작성 원칙

코드 추가/수정/삭제 또는 컨벤션 재적용으로 동작이 바뀌는 경우, 구현 전에 변경 대상의 기대 동작과 실패 케이스를 테스트로 먼저 정의한다.

1. 테스트 코드는 대상 프로젝트 내부에 작성하지 않고, 별도의 테스트 전용 프로젝트를 생성하거나 기존 테스트 프로젝트를 사용한다.
   - 테스트 프로젝트는 대상 프로젝트와 1:1로 대응되어야 한다.
   - 하나의 테스트 프로젝트가 여러 대상 프로젝트를 동시에 테스트하지 않는다.
   - 하나의 대상 프로젝트에 여러 테스트 프로젝트를 만들지 않는다.
   - 테스트 프로젝트는 대상 프로젝트를 `ProjectReference`로 참조한다.
   - 테스트 프로젝트명은 대상 프로젝트명 뒤에 `.Tests`를 붙인다.
   - 예: `Concurrency` → `Concurrency.Tests`

2. 동일한 대상 타입에 대한 모든 테스트 코드는 하나의 테스트 파일에 작성한다.
   - 테스트 파일명은 대상 타입명 뒤에 `Tests.cs`를 붙인다.
   - 예: `SerializableObject` → `SerializableObjectTests.cs`
   - 같은 대상 타입의 테스트를 여러 파일로 분리하지 않는다.
   - 파일 내부에서는 기능별 영역 또는 중첩 클래스로 테스트를 구분한다.

3. 테스트를 작성하기 위해 제품 로직을 변경해서는 안 된다.
   - 테스트 편의를 위해 실제 동작, 검증 조건, 예외 처리, 접근 제한자, 상태 변경 흐름을 바꾸지 않는다.
   - 테스트가 어렵다면 제품 로직을 왜곡하지 말고, 테스트 가능한 경계(Service, Validator, Parser 등)를 분리하는 방향을 먼저 검토한다.
   - 불가피하게 구조 변경이 필요한 경우, 변경 이유와 동작 영향이 없음을 설명하고 승인받는다.

4. 도메인 로직, 검증, 변환, 직렬화, 동시성 제어 로직은 단위 테스트 또는 통합 테스트를 작성한다.

5. 버그 수정은 수정 전에 실패하는 회귀 테스트를 먼저 작성한다.

6. 테스트 작성 후 구현하고, 구현 후 `dotnet build`, 테스트 실행, `dotnet format --verify-no-changes`를 수행한다.

7. 작업 완료 전 변경 결과에 대한 자체 리뷰를 수행하고 결과를 정리한다.
   - 변경된 코드에 `CodeStyle.md`의 규칙이 모두 적용되어 있는지 확인한다.
   - 작업 절차에 `ProjectRule.md`의 규칙이 모두 적용되어 있는지 확인한다.
   - 작업 판단과 변경 범위가 `Doctrine.md`의 원칙에 맞게 진행되었는지 확인한다.
   - 문제점
   - 우려점
   - 더 좋은 방향
   - 남은 리스크
